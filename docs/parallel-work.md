# Parallel work — low-collision focus areas

Purpose: let a second person contribute while the effects-engine work is active, without two
people editing the same hot files. Written 2026-07-20. See `docs/action-feats-handoff.md` for
the action-feats plan and `docs/effects-engine-design.md` for the engine itself.

## The collision map — hot files to avoid while effects work is active

These are the effects-engine's live surface. If effects work is in flight, stay out of these
(or coordinate explicitly and branch — see below):

- `packages/core/src/`: `foundry.ts`, `candidate.ts`, `resolution.ts`, `passive.ts`,
  `strike.ts`, `toggles.ts`, `actions.ts`, `selectors.ts`
- `apps/web/src/features/builder/rules.ts`
- `apps/web/src/features/characters/sheet/` (the sheet and its tabs)
- `apps/web/src/features/builder/data/*.json` (baked content — regenerated by
  `remap-effects.mjs`, so a hand edit is fragile anyway)
- `apps/bot/src/rules/automation.js`, `apps/bot/src/state/automation.js`,
  `apps/bot/src/commands/use/`

Everything else is fair game.

## Branching practice

`test` is the integration branch; `main` deploys (web auto-deploys from it, the bot is manual —
see `DEPLOY.md`). When two people are working at once, whoever is NOT on a given hot file should
branch off `test` and merge deliberately, rather than both committing to `test` directly. Keep
changes surgical so merges stay clean.

## The web-testing constraint (read before picking web work)

**Only the project owner can currently connect to the live Supabase**, and the web app is
auth-gated behind it. A second person therefore CANNOT, today:

- run the web app end-to-end against real data,
- do a browser verification of a sheet change (the phase-3 toggle render/persist check is
  blocked for exactly this reason),
- drain the review queue (admin-only by RLS) without being granted access.

So for a second contributor, **the best-fit web work is what does NOT need the live DB**: pure
logic, component behavior, and anything testable in isolation. And note the flip side —

> **Standing up a shared/seedable test environment is itself a high-value, low-collision task.**
> A dev Supabase (or a seeded local instance) with a couple of fixture characters would unblock
> web verification for everyone, including the browser checks nobody can currently run. It
> touches infra, not the engine. Strong candidate for a first parallel task.

The web suite is also currently **pure-logic only** — there is no jsdom / React Testing Library
setup (this bit the phase-3 work: a toggle control could be unit-tested for its collector but
not rendered in a test). Adding that infra is broadly useful and collides with nothing.

## Focus areas, by collision surface

### Tier 1 — inside the effort, near-zero code collision

- **Drain the review queue.** ~1,200 candidates + conflicts are pending human decisions; their
  effects do not ship until reviewed. This is the single biggest lever for "feats just work,"
  and it is DECISION DATA (writes to `effect_decisions`, committed separately as
  `effect-decisions.json`), touching none of the hot files. Needs admin access to the review UI.
  It is also an owner-judgment task by nature — exactly the kind of call the project owner should
  make.
- **Rules-source prep.** The eventual action-authoring pass is rules-from-source: someone has to
  assemble the rules TEXT for feat families (never model memory). Pure content, zero code
  collision, directly feeds the next big slice.

### Tier 2 — separate surfaces, parallel features

Whole subsystems with their own files, barely touching the engine:

- **Bot, non-effects**: the combat tracker (v2 is already split pure/impure), calendar/weather,
  downtime, notes, `state/snippets.js`, character management. See `apps/bot/CLAUDE.md`.
- **Web, non-sheet**: campaigns, NPCs, quests, the admin dashboard, PDF export, builder UX polish
  (portraits, drafts, recommendations). Each has its own migration/components. (Mind the
  web-testing constraint above — favor pieces verifiable without live data, or pair with the
  test-environment task.)
- **Test infrastructure**: set up web component testing (jsdom + RTL). Useful across the board,
  collides with nothing.

### Tier 3 — separable but coordinate on a boundary

- **`packages/db` wiring** — moving the web off the JSON datasets to the DB content store (built
  but unwired; the web still reads `apps/web/src/features/builder/data/`). Real, unstarted, mostly
  separable — but it touches the data the sheet reads, so agree on the `builder/data` boundary
  first. Medium collision.
- **Effects-pack ingest** (the stances / action-riders content gap) — a discrete effects-engine
  slice with a clean file boundary: `ingest-pf2e.mjs` plus a new effects-ingest module. Workable
  in parallel *if* one person owns it and the other stays out of `foundry.ts`. Needs the Foundry
  clone. This is a legitimate way to stay IN the effects engine without colliding.

## Non-negotiables (apply to everyone)

- Rules math lives in `packages/core` — never in `apps/web` or `apps/bot`. The bot is un-frozen to
  CONSUME core, not to grow its own rules.
- Rules-from-source: implement PF2e rules only from provided text, not memory.
- DB schema changes need their own plan — one Supabase project serves the live bot. Code on `test`
  is safe; a migration is not.
- Verify against the code, including the docs — every status has been wrong at least once.
